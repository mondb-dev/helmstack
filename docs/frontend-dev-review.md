# HelmStack — Front-End Development Review & Backlog

A curated, evaluated checklist for evolving HelmStack into a first-class **perception + execution layer for front-end development**.

Reviewed: 2026-06-18. Scope: `apps/desktop`, `packages/*`, `docs/*`.

**Legend** — Priority: `P0` ship-blocker / `P1` high / `P2` nice-to-have. Effort: `S` <1d / `M` ~days / `L` >1wk.

---

## 0. Strategic framing (read first)

HelmStack today is genuinely two products sharing one runtime:

1. **An autonomous web-agent substrate** — social-feed perception (`dom-extractor.ts` lines 276–917, ~640 lines), OAuth detection, encrypted account vault + TOTP, anti-detection/stealth (`anti-detection.ts`), human handoffs for captcha/2fa/payment.
2. **A front-end dev/test instrument** — console+network capture, visual diff, CWV, a11y audit, element-style inspector, component tree, responsive viewport suite, perception diff (`docs/dev-team-features.md`).

The **dev-tool half is the stated goal** and is already strong and well-architected. The agent-substrate half is largely orthogonal to front-end development and in places actively *fights* it (see §1 anti-detection jitter). This review treats the dev-tool as the product and the agent layer as the transport.

- [~] **`P1`** Decide and document the positioning. *Documented; the decision is the maintainer's.* See **[positioning.md](positioning.md)** — a grounded analysis + recommendation. Key findings: stealth/anti-detection is **already opt-in** (`HELMSTACK_STEALTH`, default off — that half of the complaint is solved); the remaining bleed is **always-on social-surface perception** (`collectSocialSurface`/`classifyPageKind` run on every page incl. `localhost`) and tool-list noise from the vault/handoff/approval surfaces. Recommendation: **capability flags within one binary** (gate social perception behind `HELMSTACK_SOCIAL` default-off mirroring the stealth pattern; register agent-substrate MCP tools only under an `agent-substrate` capability; expose `fe-dev`/`agent-substrate`/`full` profiles), deferring a package split until the flags settle the boundary. **Step 1 implemented** (maintainer chose option B): social-surface perception is now opt-in via `HELMSTACK_SOCIAL` (default off) — `runtime-config.isSocialPerceptionEnabled` + `extractPageObservation({ includeSocial })`, with the preload reading the flag; a plain web app is no longer mislabelled `social-feed`. Remaining (steps 2–4): the `agent-substrate` MCP-tool capability gate and the `fe-dev`/`agent-substrate`/`full` profiles.
